# 博尔局域网管理软件 - 离线试用授权方案

## 目标

本文档用于设计一套**不依赖在线授权服务**、适合当前阶段快速落地的**离线试用授权机制**，满足以下目标：

- 每个客户单独打包，包内携带唯一授权信息
- 默认仅允许试用 `3` 天
- 客户修改本机时间后，不能重新获得试用时长
- 客户卸载后重装，不能重新获得试用时长
- 尽量不增加客户可见复杂度
- 优先在**服务端**进行拦截，客户端只显示中文提示

## 适用场景

该方案适用于：

- 当前客户数量不多，可以接受“按客户单独出包”
- 暂时不想搭建在线授权服务器
- 希望先把试用控制做起来，再决定是否升级到在线租约模式

该方案**不是绝对防破解方案**，但对一般试用客户已经具备较好的约束效果。

## 总体思路

采用“双层授权控制”结构：

1. **包内授权文件 `license.dat`**
   - 打包时按客户生成
   - 包含客户标识、试用时长、签发时间、包编号等信息
   - 使用私钥签名，程序内置公钥进行验签
   - 防止客户直接修改到期信息或篡改客户标识

2. **本机运行状态文件 `license.state`**
   - 第一次启动时生成
   - 记录首次启动时间、最近一次启动时间、机器绑定信息
   - 用于检测时间回拨、检测重装、检测是否拷贝到其他机器继续使用

## 控制点选择

推荐将真正的授权控制放在**服务端**：

- 服务端启动时校验授权
- 授权失败则拒绝启动核心服务
- 客户端不直接做最终授权判定，只读取服务端状态并显示中文提示

原因：

- 客户端更容易被绕过
- 业务核心依赖服务端 API，服务端被锁定后，客户端无法正常使用
- 实现复杂度更低，控制效果更好

## 授权文件设计

授权文件建议命名为：

- `config/license.dat`

建议为 JSON 或二进制结构，最终外层可做 Base64 封装或简单混淆。核心字段如下：

```json
{
  "product": "boer-lan-server",
  "customer_id": "CUST_20260312_001",
  "customer_name": "某某体验客户",
  "package_id": "PKG_20260312_A1B2C3",
  "edition": "trial",
  "trial_days": 3,
  "issued_at": "2026-03-12T12:00:00Z",
  "not_before": "2026-03-12T12:00:00Z",
  "machine_bind_required": true,
  "machine_hash": "",
  "features": ["client", "server"],
  "watermark_id": "WM_20260312_001",
  "signature": "<ed25519_signature>"
}
```

### 字段说明

- `product`：产品标识，防止不同程序之间混用授权
- `customer_id`：客户唯一编号
- `customer_name`：客户名称，仅用于记录和排查
- `package_id`：本次打包唯一编号，用于识别包来源
- `edition`：版本类型，当前为 `trial`
- `trial_days`：试用天数，当前固定为 `3`
- `issued_at`：授权签发时间
- `not_before`：最早允许使用时间
- `machine_bind_required`：是否绑定机器
- `machine_hash`：如果是预先绑定机器，可提前写入；若首次启动绑定，则初始为空
- `features`：后续可扩展的功能开关
- `watermark_id`：用于识别泄露包来源
- `signature`：授权文件签名

## 本机状态文件设计

本机状态文件建议命名为：

- `%ProgramData%/BoerLAN/license.state`

同时建议写入第二份冗余副本，例如：

- 注册表：`HKCU/Software/BoerLAN/license`
- 隐藏文件：`%AppData%/BoerLAN/.license.state`

状态文件示例：

```json
{
  "package_id": "PKG_20260312_A1B2C3",
  "customer_id": "CUST_20260312_001",
  "machine_hash": "6d2f...",
  "first_seen_at": "2026-03-13T02:18:10Z",
  "last_seen_at": "2026-03-15T08:03:21Z",
  "launch_count": 17,
  "time_rollback_detected": false,
  "state_checksum": "<sha256_or_signature>"
}
```

### 状态文件作用

- 记录首次启动时间，作为试用期起点
- 记录最近一次启动时间，用于检测时间回拨
- 记录机器摘要，防止复制到其他机器继续使用
- 记录包编号，防止不同包之间相互覆盖试用状态
- 通过多处冗余保存，降低卸载重装后重置试用期的可能性

## 机器绑定设计

建议生成 `machine_hash` 时使用多个本机信息组合后做哈希，例如：

- Windows `MachineGuid`
- 系统盘卷序列号
- 主机名
- CPU / 主板信息（可选）

建议流程：

1. 读取多个系统稳定标识
2. 拼接为统一字符串
3. 使用 `SHA-256` 计算摘要
4. 最终得到 `machine_hash`

说明：

- 不建议只用网卡 MAC，变化太频繁
- 不建议绑定过多易变信息，否则客户正常换配置后也会失效
- 当前阶段建议优先使用：`MachineGuid + 系统盘卷序列号 + 主机名`

## 防时间回拨策略

单纯检查“当前时间是否超过 3 天”是不够的，必须增加防时间回拨策略。

建议规则如下：

1. 第一次启动时记录：
   - `first_seen_at`
   - `last_seen_at`

2. 每次启动时检查：
   - 当前时间是否早于 `last_seen_at`
   - 如果早于，并超过容差阈值（建议 `10` 分钟），视为时间回拨

3. 一旦发现时间回拨：
   - 标记 `time_rollback_detected = true`
   - 直接判定授权失效，拒绝继续启动

### 建议判定逻辑

- `now < last_seen_at - 10分钟` → 判定时间回拨
- `now > first_seen_at + 3天` → 判定试用过期
- `machine_hash != state.machine_hash` → 判定机器不一致

## 防卸载重装策略

卸载重装后重新获得试用期，通常是因为授权状态只保存在程序目录下。

为避免这一问题，建议：

- 状态文件保存到 `ProgramData`
- 再保留一份注册表副本
- 卸载程序时不主动删除授权状态
- 重新安装时优先读取历史状态

这样即使客户卸载并重新安装，只要没有手动清理多个状态位置，仍会恢复原来的试用起点。

## 签名与加密策略

### 必须做：签名

推荐使用：

- `Ed25519` 非对称签名

设计原则：

- 私钥只保存在打包机或内部签发工具中
- 程序内只放公钥
- 程序启动时对 `license.dat` 验签
- 验签失败即视为授权文件无效

### 可选做：内容混淆/加密

为了避免客户一眼看懂授权文件内容，可以额外做：

- Base64 包装
- JSON 压缩
- 二进制序列化
- 简单对称加密后再存储

但需要明确：

- **安全性主要来自签名，不是加密**
- 即使不做加密，只做签名，也已经足够防止普通篡改

## 启动校验流程

推荐服务端启动前或启动最早阶段执行如下逻辑：

```text
1. 读取 license.dat
2. 使用公钥验签
3. 读取或创建 license.state
4. 计算当前 machine_hash
5. 若首次运行：写入 first_seen_at / last_seen_at / machine_hash
6. 若非首次运行：
   - 检查 machine_hash 是否一致
   - 检查当前时间是否回拨
   - 检查是否已超过 3 天
7. 任一检查失败：拒绝启动核心服务
8. 全部通过：更新 last_seen_at / launch_count
```

## 客户端显示策略

客户端不直接做授权判定，仅做中文提示展示。

建议统一为较弱提示，不暴露内部控制细节，例如：

- `服务初始化失败，请联系供应商`
- `服务暂不可用，请联系供应商`
- `试用已结束，请联系供应商`

不建议直接显示：

- `授权签名失败`
- `检测到时间回拨`
- `机器绑定不匹配`

原因是避免客户过早意识到具体防护逻辑。

## 打包流程建议

建议未来按以下流程为每个客户出包：

1. 输入客户编号、客户名称、试用天数
2. 生成唯一 `package_id`
3. 生成 `license.dat`
4. 使用私钥签名
5. 将 `license.dat` 复制到服务端包内固定目录
6. 重新打包便携版
7. 记录 `customer_id / package_id / watermark_id`

## 推荐目录结构

服务端最终便携包示例：

```text
Boer-LAN-Server/
├── Boer-LAN-Server.exe
├── backend-server.exe
├── config/
│   ├── config.yaml
│   └── license.dat
└── 启动说明.txt
```

## 最小可实施版本（V1）

为保证当前阶段快速落地，建议只做以下内容：

- 服务端读取 `config/license.dat`
- 使用公钥验签
- 首次启动写入 `license.state`
- 绑定机器码
- 防时间回拨
- 试用期固定 3 天
- 授权失败时服务端拒绝启动
- 客户端显示统一中文提示

该版本已经能覆盖：

- 客户改系统时间
- 客户卸载重装
- 客户复制包到另一台机器

## 后续可扩展版本（V2）

后续若需要更强控制，可扩展：

- 在线授权租约
- 后台吊销客户包
- 远程停用
- 功能开关分级
- 正式版长期授权
- 自动续期

## 风险与边界

需要明确，本方案的安全边界如下：

- 对普通客户和一般试用场景，控制效果较好
- 对具备逆向能力的高级用户，不能保证绝对不可破解
- 不联网的前提下，不存在 100% 无法绕过的本地授权系统

因此该方案的定位应是：

- **当前阶段最省事、最实用、最容易落地的试用控制方案**
- 后续若业务规模扩大，再升级为在线授权体系

## 结论

当前阶段推荐采用：

- **每客户单独打包**
- **包内签名授权文件 `license.dat`**
- **本机状态文件 `license.state`**
- **服务端启动校验**
- **防时间回拨**
- **防卸载重装**
- **默认 3 天试用期**

这是在“不想太复杂、不想搭云授权服务、但又想尽量防试用白嫖”的前提下，最合适的折中方案。
